What this redesign does

Attorneys landing on /legal/ai-assistant/templates today see a flat table with two small action buttons top-right ("Import Templates" and "New Template") that are visually equal to the search field — so the highest-value action (importing their existing library of DOCX/PDFs in one pass) gets buried. This redesign promotes both entry points to a hero pair, adds category-colored accents so PI / Family / Immigration templates are scannable at a glance, adds a jurisdiction filter (for the matrix), and modernizes the card chrome.

Parity with LegiDraft. The LegiDraft redesign introduced a "two-CTA empty-state" pattern (New Draft · Use Template). This mockup mirrors that idea for the Template Library: a big split-hero where Import is the primary action (because onboarding a firm's existing templates unlocks ~80% of the value in one session) and Create from scratch is the secondary action (ongoing, occasional).
Phase A Essentials

Hero CTAs, jurisdiction filter, Save-as-Template modal with variable detection, Share/Export cleanup. 2 days.

Phase B Visual polish

Per-category color accents, per-doc-type icon map (extract from template-picker-inline.component.ts), stat strip, chip-based active filters. 1 day.

Phase C Extras

Optional Recently Used strip, loading skeleton with shimmer, filtered empty state. 0.5 day (or punt).

What's wrong with the live page today

From the screenshot of the production page at /legal/ai-assistant/templates.

Live today — pain points
  • Primary actions buried. "Import Templates" + "New Template" are 140-px buttons lost in the top-right; the second one is literally cut off when a modal overlays.
  • Single "Total: 17" stat doesn't tell the attorney how many are theirs vs. imported vs. org-shared vs. private.
  • No jurisdiction filter — but PA × jurisdiction matrix exists on the backend (Sprint 1). Cards can't be filtered by it.
  • Flat table with generic file icons. Can't scan for "all Immigration templates" without reading every row.
  • No visual indication of ownership. Who owns each template? Yours? Imported? Private-to-you?
  • Share / Export buttons in the row-level menu are placeholders with no real backend — dead UI.
After this redesign
  • Split-hero with illustration promotes Import + Create to first-class citizens.
  • 4-tile stat strip (Total · Yours · Imported · Private) with trend indicators.
  • Jurisdiction select joins Category + Practice Area + Sort (new-badged on first render).
  • Category-colored card bands (3px top accent) + per-doc-type icons make scanning O(1).
  • Ownership tag top-right of every card: YOURS / ORG / IMPORTED / PRIVATE.
  • Share / Export removed — replaced by real "Use" + "More" (rename, duplicate, delete).

Why one editor, two entry points. The "New template" button (page header) and "Save as Template" modal (generated draft toolbar) end up at the same place: a CKEditor body + metadata sidebar + live variable panel. The difference is pre-fill — one starts blank, the other starts with the generated HTML + auto-detected variables — but everything downstream (validation, auto-save, save button) is identical.
Live variable panel is the killer feature. As soon as the attorney types {{anything}}, the right rail picks it up. No "run variable detection" step, no modal. This is what makes a static legal document feel like a reusable asset — you see the parameters accumulating in real time.

Scope

What this redesign does NOT do

🚫 Not in this redesign:
  • No new backend endpoints — wire existing /api/ai/templates/* and /api/ai/templates/import/* as-is.
  • No changes to AILegalTemplate or AITemplateVariable entities.
  • No new Flyway migration.
  • No Share/Export button placeholders — both removed (replaced by real "Use" + kebab with rename/duplicate/delete).
  • No in-page Collections / Folders grouping — punt to a later sprint.
  • No bulk selection / bulk export — single-item actions only.
  • No changes to the 4-step Import Wizard's backend behavior (Sections 7 is a visual/UX sweep over the already-shipped flow — analyze/poll/review/commit endpoints are untouched).
  • No Claude-model change — Sonnet for first analysis, Haiku for re-analysis stays as shipped.
  • No .legidraft-skin wrapper class — brand tokens are defined inline and work with or without a skin.
Open questions before coding:
  1. Share/Export buttons — mockup shows them removed. Confirm deletion, or keep them as ghosted placeholders?
  2. New-template editor entry (Section 8) — same CKEditor component as DocumentEditorComponent, or a stripped-down variant without AI rewrite toolbar? Recommendation: reuse verbatim, hide the AI-rewrite buttons via an @Input() mode: 'draft' | 'template'.
  3. Phase scope — Phase A (filters + save-as modal + jurisdiction select), A+B (add category colors + page-header redesign), or A+B+C (add Recently Used strip + import-wizard polish)?
  4. Recently Used horizontal strip — worth a second fetch /api/ai/templates?sort=lastUsed&limit=6 on page mount, or skip for v1?
