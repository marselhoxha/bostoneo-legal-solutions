LegiDraft · Flow v2 · proposal · 2026-04-18

One workspace. Two paths. Zero modals.

The current LegiDraft stuffs a page-sized Template Library into an NgbModal, then opens a second modal for the auto-fill wizard. It also duplicates the left sidebar by adding a LegiDraft-specific rail next to the legacy one. This proposal: the legacy sidebar stays shared at the arrival state of every task, but collapses away the moment the user commits to a draft flow — AI wizard, template picker, fill & generate, or editor all get the full width. One pane, two paths, no modal-on-modal.

01

Problem — current flow regressions

2 bugs · 3 smells

The LegiDraft flow shipped in Sprint 4a/4b works, but it has two regressions and a few design smells that are eroding trust in the layout.

Bug 1 · Duplicate sidebar. Clicking LegiDraft mounts the new DraftHistoryRailComponent (280px) next to the legacy .conversation-sidebar. Two "Recent Drafts" rails appear side-by-side. Root cause: we added the new rail without removing the old one.
Bug 2 · "Link a case" is a no-op. The Case Context empty-state CTA is wired to startNewDraft(), which opens the wizard at Step 1. The user has no way to just pick a case from the right rail. FIXED in 4a-HF2
Smell 1 · Modal-on-modal. "Use Template" opens template-library-embedded-modal containing a page-sized TemplateLibraryComponent. Clicking Use on a template then opens a second NgbModal with the 4-step AutoFillWizardComponent. Right-rail Case Context disappears during the whole dance.
Smell 2 · Two parallel wizards. DraftWizardComponent (AI path) and AutoFillWizardComponent (template path) collect overlapping info (case, client, variables, output format) in incompatible shapes. No shared convergence step.
Smell 3 · Sidebar inconsistency. Questions tab has one sidebar layout; Draft tab has a different one. Consistency is the contract we made with the user — we're breaking it.
02

Three options considered

A · B · C

I considered three distinct flows before committing. All three preserve the sidebar contract. They differ in how the center pane is structured and how templates become discoverable.

Option A · Unified Inline Workspace

Draft Dashboard lands as two tiles; both AI-draft and template paths morph the same center pane; they converge at a shared Fill & Generate step. No modals.

  • Both wizards converge → less code duplication
  • Reuses Sprint 4a/4b components with minimum rework
  • Slightly more state in ai-workspace (one extra view mode)

Option B · Command Palette

A single prompt bar ("What do you want to draft?") with inline autocomplete that mixes AI-draft suggestions and matching templates. Power-user pattern.

  • Fastest for power users
  • Minimal visual footprint
  • Discoverability-poor for new attorneys
  • Requires typing skill before any visual scaffolding
  • Templates buried one level deep

Option C · Tabbed Workspace

Tab strip at the top of the draft pane: [AI Draft] [From Template] [Recent] [All Templates]. Flat, each tab is an inline pane.

AI Draft Pane

  • Flat; every feature one click away
  • No modals
  • Loses the "arrival moment" that orients attorneys
  • [All Templates] tab duplicates /legal/ai-assistant/templates
  • Tabs imply orthogonality, but the flows actually converge
Why Option A wins. It fixes both regressions in one move, gives templates first-class surface area, merges the two wizard code-paths at the convergence step, and reuses the existing Sprint 4a/4b components (empty-state → dashboard, wizard stays, case-context-panel stays). Option B is clever but discoverability-poor; Option C is flat but loses the arrival moment and duplicates the standalone template library route.

The arrival state replaces the current single-card empty state. Two equal-weight tiles signal the two valid paths; a "pick up where you left off" recent-drafts strip below gives attorneys a third door for unfinished work.

  • Fraunces serif headline for editorial confidence (distinguishes from grid CTAs)

Recent drafts strip
  • Three most-recently-edited drafts across both AI and template sources
  • Each card shows: document type eyebrow, case/title, edit timestamp, completion state
  • Click → reopens that draft in the editor directly (skips the whole flow)

Replaces: the current single-card draft-empty-state with its two buttons. That component becomes a thin wrapper around the new dashboard layout — same file, bigger canvas.
Task-card row above dashboard
  • Compact version of the 4 task cards stays visible at the top of the draft pane
  • Active state shows which mode is selected
  • Clicking another task (e.g. Question) swaps the center pane to that task's arrival; the left rail's content also swaps to match (Recent Q&A / Workflows / Uploads)

08

The sidebar contract

home shows · flow hides

The legacy .conversation-sidebar is the shell. Its content swaps per active task, but the shell stays identical across every LegiSpace task. Within LegiDraft only, the sidebar hides once the user commits to a draft flow (AI wizard, template picker, fill & generate, generating, editor) — giving the wizard / picker / editor the full horizontal width. For Question, Workflow, and Upload tasks the sidebar stays visible throughout.

Template path · Dashboard → click "Draft from a template" → Template Picker (sidebar ✗) → Fill & Generate (sidebar ✗) → Editor (sidebar ✗). Same rule: sidebar visible only at Dashboard.

Back-to-dashboard · Clicking "← Back to dashboard" in the inline back-bar at any state returns to the Dashboard; the sidebar slides back in.

Switching tasks · The compact task-card row at the top of the center pane remains visible so the attorney can jump to Question / Workflow / Upload at any time; doing so returns the sidebar.
Why hide the sidebar during the flow? The wizard's 5-column practice-area grid, the template picker's 3-column card grid, and the Fill & Generate 2-column form all need horizontal room. Keeping the 280px sidebar visible compresses those grids to the point of feeling cramped. The Case Context right rail stays because that's the high-value reference during drafting — linked case info, dates, insurance. Asymmetric collapse (hide left, keep right) directs focus to the work AND to the case it's grounded in.
Reverting Sprint 4a-HF1 / deleting Sprint 4b rail. Sprint 4a-HF1 added && selectedTask !== 'draft' to the legacy sidebar's *ngIf — that gets replaced with a more precise guard: sidebar stays visible when selectedTask === 'draft' && draftMode === 'dashboard', hides when draftMode is any non-dashboard state. The DraftHistoryRailComponent created in Sprint 4b is deleted; its job moves back into the legacy sidebar's draft-dashboard content branch. The .legidraft-sidebar-left CSS is deleted.

Risks, open questions, order of work

pre-implementation
Risks
R1. Deleting AutoFillWizardComponent affects /legal/ai-assistant/templates (standalone template library). That page calls useTemplate() which opens the auto-fill wizard modal there too. We need to either keep AutoFillWizardComponent alive for the standalone page, or port it to use FillAndGenerateComponent inline as well. Recommendation: port both paths to FillAndGenerateComponent, delete AutoFillWizardComponent.
R2. Variable auto-fill from case context needs the full case loaded before rendering. We need to guard the Fill & Generate view until case$ resolves (not just the id). Existing Sprint 3 fix #101 for hydrateFromCase is analogous.
R3. Inline template picker can't re-use TemplateLibraryComponent directly — its page-level chrome doesn't fit. The new picker is a leaner component. CRUD (create/edit/delete templates) stays at /legal/ai-assistant/templates.
Open questions
Q1. Should the primary tile text be "Start a new draft" or "Draft with AI"? The latter is clearer about the mechanism; the former is more inviting. Leaning toward "Start a new draft" with the AI · guided flow eyebrow doing the disambiguation work.
Q2. Should the secondary tile show 4 recent templates or 4 case-matching templates? Recent = familiar; case-matching = contextual. Recommend: case-matching when a case is linked; recent otherwise.
Q3. On the template picker, should a template without enough variables to auto-fill still be suggested? Yes — surface with a warning chip ("2 of 6 vars autofilled") so the attorney knows they'll fill more manually.
Q4. Where does "Save as template" live? Keep in editor toolbar post-generation — unchanged from Sprint 2.
Implementation order (post-approval)
  1. Revert Sprint 4a-HF1 sidebar guard. Remove && selectedTask !== 'draft' from legacy sidebar's *ngIf. Restore DRAFT MODE CONTENT block with proper Recent Drafts shell.
  2. Delete DraftHistoryRailComponent. Remove folder, remove import, remove from .legidraft-focused.
  3. Refactor draft-empty-state → draft-dashboard. Two tiles + recent-drafts strip.
  4. Build template-picker-inline. Filters + grid + search. Leaner than TemplateLibraryComponent.
  5. Build fill-and-generate. Flattened single-page form. Ports AutoFillWizardComponent's variable logic.
  6. Add view-mode state to ai-workspace. draftMode enum drives which component renders in center.
  7. Wire inline back-bar. Shared across all morphed states.
  8. Delete AutoFillWizardComponent + port standalone template library's useTemplate() to use the new inline components.
  9. Delete ng-template #templateLibraryModal + CSS. No more template modal.
  10. E2E verification. Both paths, on a real MA PI case, with and without linked case. Dark mode. Mobile/tablet.
Approval gate. Sprint 4c starts only after this preview is approved. The current Sprint 4a/4b work stays on develop untouched; the new work adds on top rather than replacing piecemeal.
