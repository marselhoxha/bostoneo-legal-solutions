The Problem

Two compounding failures today

Attorneys stop editing case data because the interaction is painful, and the PI insurance layout looks broken when one side has no data.

Failure 1 — Global Edit Mode.
Scroll to top → click "Edit" → scroll back down to the field → change it → scroll back up to Save. On a ~3000-line page with 12+ sections, this is painful enough that attorneys resort to emailing paralegals to update fields for them.
Failure 2 — Empty sections render as hollow cards.
PI "Client's Insurance" next to a populated "Defendant's Insurance" shows as a card with just a header and empty body — looks like a bug, not a blank form.

Section Edit Modal

One reusable modal component for every section

Velzon NgbModal size lg, centered, static backdrop (backdrop click does nothing — ESC or explicit Cancel only). Mirrors the TaskAssignmentModalComponent pattern already used throughout the codebase.

Modal — "Edit Client's Insurance"
Edit Client's Insurance (for PIP / UIM)
Please enter a valid email address.
Partial PATCH save. Only dirty form controls are sent to PATCH /api/v1/legal-case/update/:id. If the attorney only touched "Adjuster Email", the body is { clientInsuranceAdjusterEmail: "..." }. Audit log stays clean, concurrent edits to other fields don't clobber.
Cancel with unsaved changes. Clicking Cancel or ESC with a dirty form fires a SweetAlert2 "Discard changes?" confirm. Backdrop is static so an accidental outside click does nothing — no lost work.
On save success. Modal closes, toast confirms, the parent section card updates in place from the PATCH response payload. No GET /case/:id reload — scroll position preserved exactly.
On save failure. Spinner stops, red alert appears inside the modal body, the modal stays open, user's edits remain in the form for retry.

Phased Rollout

6.5 half-days — each phase ships independently

The global Edit button stays in place until Phase 4. Any phase can be reverted without touching the others.

Phase 1a

Modal scaffold + PI insurance sections

Build CaseSectionEditModalComponent. Wire to Defendant's and Client's Insurance. Implement empty-state + paired-row rendering.

~2 half-days
Phase 1b

Remaining PI sections

Injury Info, Defendant Info, Accident Details — config-only once the modal is generic.

~0.5 half-day
Phase 2

Other practice areas

Criminal, Family, Immigration, Real Estate, IP — same code path, different config.

~0.5 half-day
Phase 3

Core sections + backend gap

Client Info, Court Info, Important Dates. Extend patchCaseFields() to cover dates/court/billing fields currently dropped.

~1.5 half-days
Phase 4

Delete global Edit mode

Single-field mode for Title/Description/Status/Priority. Rip out isEditing, saveCase(), edit-mode template, editForm. ~250 lines removed.

~1 half-day
Phase 5

Polish

Empty-state icons, post-save flash, cancel-confirm, mobile pass, docs.

~1 half-day
Zero new backend for Phases 1–2. PATCH endpoint LegalCaseResource.java:277-291 and frontend caseService.patchCase() already exist but are unused today. Phase 3 is the only phase that touches the backend (~20 lines added to patchCaseFields() for dates/court/billing/client-insurance fields).
Out of Scope

Explicitly NOT doing

  • Optimistic UI updates before server confirms.
  • Concurrency control (ETag / @Version) for two-tab editing — known limitation, last-write-wins.
  • CanDeactivate route guard for unsaved modal state.
  • Rich-text editor for description.
  • Keyboard shortcuts (Cmd+E etc).
  • Audit-log viewer (who/when changed each field).
  • Field-level inline diff/history.
  • Breaking the 3000-line case-detail.component.ts into child components — we shrink it by ~250 lines, that's the win for this sprint.
  • Documents / Notes / Timeline / Events / Team tabs — unchanged.
  • Billing card visual redesign — only adds PATCH support.
  • Mobile FAB "+" for quick add.
  • Undo toast after save.
  • Custom practice-area field creation by admins.
