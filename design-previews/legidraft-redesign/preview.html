What we're solving

After deep codebase research, I discovered the template system already exists (AILegalTemplate, TemplateLibraryComponent, AutoFillWizardComponent) but is hidden from users. I also found a deeper problem: every AI-generated document cites Massachusetts law regardless of the selected jurisdiction, because the 17 prompt templates hardcode MA statutes. The real work is integration + a wizard for the AI path + a practice-area × jurisdiction matrix so an attorney in Texas gets Texas law and an attorney in Family Court gets divorce-petition structure.

The Matrix Problem. Legal writing is deeply state- and practice-area-specific. Motion to Dismiss in MA cites Mass. R. Civ. P. 12(b)(6), in TX it's Rule 91a, federally it's Fed. R. Civ. P. 12(b)(6). Family Law uses "Petitioner/Respondent" not "Plaintiff/Defendant". Today the AI ignores this and produces MA/federal-flavored output for every case — see the Matrix section for the fix.

Multi-purpose LOR, client insurance fields, + 4-way template fallback cascade, single source-of-truth enums, MA/Federal citation packs. 4-5 days.

Foundation

Practice Area × Jurisdiction Matrix

The core design decision that makes LegiDraft state-aware and practice-area-aware. All three sprints build on this.

The problem today

17 JSON prompt templates
Only 1 is state-specific (motion-texas.json). The other 16 hardcode MA law.
2-way fallback
{type}_{state} → {type}. No practice-area branch. No citation packs.
System prompt
Zero instructions about state law or practice-area conventions.
Consequence. An attorney creates a PI case with jurisdiction: "Texas" and asks for a motion. The AI cites Mass. R. Civ. P. 12(b)(6) because that's what the template says. The Texas rule is 91a. The document is wrong at the top of page 1.

The fix: 4-way template fallback cascade

When the AI needs a template, the registry tries the most specific filename first, then falls back step-by-step. Launch covers ~15 hand-authored templates (MA + Federal × PI, Family, Criminal) — everything else uses fallback.

  • ~15 PA × state templates (PI, Family, Criminal in MA + Federal)

1B. Multi-Purpose LOR — API Contract

API Contract — POST /api/v1/ai-workspace/init-draft-conversation
BEFORE (today)
// Old LOR — always PI liability, policy limits only
{
  "caseId": 4421,
  "prompt": "draft LOR for Doe v. Smith",
  "documentType": "letter-of-representation",
  "jurisdiction": "Massachusetts"
  // NO practiceArea — registry can't route by PA
  // NO way to specify recipient or purposes
}
AFTER (with practiceArea + documentOptions)
// New — PA-routable + multi-purpose
{
  "caseId": 4421,
  "prompt": "draft LOR for Doe v. Smith",
  "documentType": "lor",
  "practiceArea": "pi",        // NEW
  "jurisdiction": "Massachusetts",
  "courtLevel": "DEFAULT",
  "documentOptions": {     // NEW
    "recipientType": "client_insurer",
    "purposes": [
      "representation_notice",
      "pip_application_request",
      "uim_claim_notice"
    ]
  }
}
Registry resolution trace. With the new payload, DocumentTypeTemplateRegistry.getTemplateText("lor", "pi", "ma") runs the 4-way cascade and hits lor_pi_ma.json at level 1. Then it loads the MA jurisdiction pack from jurisdictions/ma.json. Then it prepends a "JURISDICTION CONTEXT" section to the prompt listing MA rules + common statutes. The AI sees everything it needs to cite M.G.L. c. 175 § 112C correctly.
Why documentOptions is a Map<String, Object>: Every document type has different config needs. LOR has recipient+purposes. Motion has grounds+court level. Interrogatories has count+topics. A flexible map lets each type define its own schema without requiring backend changes when we add new types. The lor_pi_ma.json template file defines what to do with these options.

1.5G. Edge Cases Handled (All in MVP Scope)

Scanned PDF
PDFBox returns < 100 chars on a multi-page PDF → reject with "OCR not supported yet — convert to text and re-upload". No silent degradation to empty template.
Password-protected file
Catch EncryptedDocumentException / InvalidPasswordException → 422 with friendly message.
Macros in .docx
VBA stripped for safety. Text extracted normally. Warning surfaces in review.
PII baked into template
Named party repeats ≥ 3× → flag as likely real client info with "Convert to variable" CTA.
Duplicate detection
SHA-256 of extracted text matched against org's existing templates. Skip / Overwrite / Create Anyway.
Low confidence (< 60%)
Don't auto-classify. Attorney picks practice area + jurisdiction manually from dropdowns.
5-min session TTL
Attorney can review within 5 min. Abandonment returns 404 on commit — forces clean re-analyze.
Non-English content
If < 50% English tokens, flag warning: non_english_content. MVP is English-only.

Practice Area drives everything. Step 1 auto-pre-selects the open case's practiceArea (e.g., pi). That selection filters Step 2's document list to just PI types, and combines with Step 3's jurisdiction selector to resolve the correct template (lor_pi_ma.json) and citation pack. Step 3 is auto-skipped for simple types (Legal Memo, Email) — user flows straight to Step 4.

Why practice area first, not category: The old design asked "Pleadings vs Motions vs Letters" — a structural taxonomy. But PI, Family, and Criminal attorneys think in practice areas, and the template cascade routes by PA. Starting with PA also lets Step 2 filter document types by what's actually relevant (an Immigration attorney doesn't need to scroll past Family Law forms). The 15 areas come from legal-constants.ts — a new single source of truth that replaces 3 duplicated lists in the current codebase.

  • Cascade resolved lor_pi_ma.json at level 1 — full hand-authored MA PI template
  • MA citation pack injected → AI knows M.G.L. c. 175 § 112C is the policy-limits statute

What the matrix did for the TX output
  • Cascade fell through lor_pi_tx (not seeded) → lor_tx (not seeded) → hit lor_pi.json (generic PI)
  • TX citation pack injected → AI substitutes Tex. Ins. Code § 541.060, uses "Texas Insurance Code" wording

This is the redesign's north star. Today, the TX version of this letter would cite M.G.L. c. 175 § 112C — wrong at the top of page 1. After the matrix, it cites the Texas Insurance Code. That's the difference between a TX attorney hand-editing every AI draft vs. signing it and sending it.

Risks & Notes

Known risks and decisions

Risk log

1. 12K-line god component. ai-workspace.component.ts is massive. Injecting the wizard must be done as a sub-component (DraftWizardComponent) that replaces only the draft-creation section. Do not add wizard logic inline — it would make the god component worse.
2. State dual-life. Current code has selectedDocTypePill (property) and selectedTask$ (observable). Wizard should use a new, dedicated state service (DraftWizardStateService) to avoid adding to the existing mess. After wizard completes, hand state over to the existing generateDocumentFlow().
3. Legacy dead code. documentTypes array at lines 843–1100 of ai-workspace.component.ts is unused (~50 config objects). Remove in Sprint 4 cleanup.
4. Encrypted field gotcha. The 5 new client insurance fields must use @Convert(converter = EncryptedStringConverter.class) — except clientInsuranceCompany which is searchable and unencrypted (mirrors existing defendant pattern).
5. Template detection accuracy. "Save as Template" v1 uses simple regex to detect {{variables}} in the HTML. Users must manually wrap text. v2 (post-launch) could use AI to suggest likely variables — not in initial scope.
6. Two wizards coexisting. AutoFillWizardComponent (template path) and the new DraftWizardComponent (AI path) are separate components. Both use the same CSS stepper pattern so they feel consistent. Users see which path they're on via context — modal (template) vs. inline (AI).
7. Template explosion risk. Full matrix = 15,900 combos. Mitigation: launch with ~15 hand-authored templates + 5 jurisdiction packs; 4-way cascade covers the rest with graceful degradation. Add 5 templates per sprint based on usage telemetry. We accept that unseeded PA × state combos produce generic-but-correct output (no hallucinated statutes) instead of state-specific output.
8. Practice-area enum duplication. Three separate lists today: case-create.component.ts (12 entries), template.service.ts (10), practice-area-fields.config.ts (6). Values drift. Sprint 1 consolidates into legal-constants.ts (15 entries) + Java PracticeArea enum. All existing call sites migrate; no fallback path — single source of truth or nothing.
9. Jurisdiction name vs. ISO-code mismatch. LegalCase.jurisdiction stores full state names ("Massachusetts"). organizations.state stores ISO codes ("MA"). Template filenames use ISO. Registry normalizes via JurisdictionResolver.getStateCode(...) at every lookup. Risk: typos in case data ("Massachusetsts") would fail resolution — add validation on case save.
Design decision — practice area filter: Currently selectedPracticeArea in LegiSpace does nothing. The wizard's Step 1 replaces this dead variable with real state. After wizard completes, PA + jurisdiction flow into the API payload and into the registry lookup — every downstream decision uses them.
